### compression research track (parallel with run sequence)

the correction-field memory design (`wiki/synthesis/correction_field_memory.md`) is the project's candidate for the next compression method. it compounds two of the six biological compression mechanisms identified in `wiki/synthesis/compression_beyond_quantization.md` — predictive filtering (mechanism 1) and generative replacement (mechanism 2) — and introduces a third effect specific to the design: because the stored values are prediction residuals rather than raw representations, they have smaller expected norm, increasing the number of distinct associations the state can hold before interference saturates. the design does not require any new architectural components beyond a lightweight prediction head per layer (~1% parameter overhead).

the compression track is validated via cpu simulation before entering the run sequence:
1. **correction-field capacity simulation**: the prompt lives at `simulations/prompts/correction_field_capacity.md`, and the checked-in harness now lives at `simulations/memory/correction_field_capacity.py`. do not treat that harness as available evidence until it has a recorded run artifact and the required prosecutor review.
2. **if simulation confirms capacity increase**: the correction-field design enters the run sequence at the slot currently held by "value-side compression" (run 2 in `blueprint.md`). it replaces rate-coded value sparsification with prediction-residual value storage.
3. **if simulation shows no capacity increase**: the correction-field design is ruled out for the current matrix memory, and the project investigates whether the capacity ceiling is a fundamental property of the outer-product mechanism or addressable by other means.

compression is a primary research direction for the project, not a secondary optimization. the six biological mechanisms compound to 10^10-10^12x effective compression in biological memory; the project's matrix memory currently implements zero of the six. the correction-field design is the first step toward compounding them.

## open research questions

1. does the architecture's asymmetric outer-product matrix memory support content-addressable retrieval at any encoding?
2. if yes, which encoding family preserves the most retrievable structure?
3. does the optimal encoding for the key half differ from the optimal encoding for the value half? biological cortex pattern-separates keys aggressively (~5% sparse) and stores values densely. does the project's architecture want the same asymmetry?
4. after the standalone overwrite sweep at `decay=0.90` showed that erasure hurts every tested encoding at the first useful 32-pattern knee, does training ever create a regime where overwrite becomes net helpful once keys remain dense?
5. the standalone decay sweep now shows that retention matters sharply: exact-query recall at `d_head=64` first reopens at 32 patterns around `decay=0.90` and at 64 patterns around `decay=0.95`. can the trained model sustain a similarly slow retention regime without collapsing optimization or saturating the state?
6. is there a structural change to the architecture that would allow a different memory operation to handle retrieval, while the matrix memory handles a different role (e.g., short-term smoothing)?
7. once we have a validated encoding from simulation, does it scale to the h200 training environment without losing its property — i.e., does standard gradient descent at 283m params find the encoding's predicted operating point?
8. does storing prediction residuals instead of raw values in the matrix memory increase effective capacity? the correction-field design (`wiki/synthesis/correction_field_memory.md`) predicts that residuals are smaller than raw values when prediction quality is nontrivial, so the state accommodates more distinct associations before interference saturates. the prompt for the planned simulation lives at `simulations/prompts/correction_field_capacity.md`; this is still a planned evidence path, not a checked-in harness.
9. once the correction-field capacity harness exists and runs, does its measured capacity gain (mean cosine recall above 0.5 threshold) increase monotonically with prediction quality q across the sweep {0.0, 0.3, 0.5, 0.7, 0.9, 0.95, 0.99}? if the capacity gain is flat across prediction quality, the self-compression property is not operating and the design reduces to a different-basis raw storage with no adaptive advantage.
10. does surprise-gated writing reduce write frequency without hurting recall on the tokens that matter? the predictive-filtering mechanism should cause most predictable tokens to skip the state update entirely, leaving the state less crowded and better able to hold the rare surprising content.
11. does the correction-field design shift the decay tolerance downward, making lower decay values viable for the same pattern count? this would partially resolve the finding that exact-query recall at d_head=64 only reopens at decay=0.90 for 32 patterns.
12. does a cross-layer depth gradient emerge in the correction-field design? the falsifiable form: at convergence, the frobenius norm of the correction-field state at layer L, normalized by the frobenius norm of the corresponding raw-value state at the same layer, should decrease monotonically with L. if confirmed, this creates a naturally emergent compression hierarchy within the existing 24-layer stack without explicit tier boundaries.

## glossary

this is the project's own vocabulary. every term here describes a thing the project is building, not a thing imported from a published paper. each entry maps a project term to its mathematical definition. adding a new term to the glossary is part of the prosecutor cycle.

- **architecture**: the neural model the project is building. has no external name. it is a sequence-processing model with two layer kinds (the matrix memory and the compressed attention) interleaved at a 6 to 1 ratio. byte-level vocabulary, 1024-dimensional residual stream, 28 layers (24 matrix-memory layers and 4 compressed-attention layers).
- **matrix memory**: the per-layer state-keeping operation `S_t = a * S_{t-1} + b * k_t * v_t^T`, optionally with erasure `S_t -= a * b * k_t * (k_t^T * S_{t-1})`. shape `(batch, num_heads, head_dim, head_dim)`. read via `o_t = q_t^T * S_t`. exists in 24 of the 28 layers.
- **compressed attention**: the per-layer operation that projects the residual stream to a low-rank cached representation per token, then computes scaled dot-product attention over the cache with rotary positional encoding concatenated onto the keys. exists in 4 of the 28 layers.
- **decay coefficient `a`**: the per-step retention factor in the matrix memory's update. computed at the start of each forward pass as `exp(logsigmoid(alpha_log_base))` for the static case, or with an optional adaptive subtraction term for the activity-dependent case. range (0, 1).
- **write gate `b`**: the per-step write strength in the matrix memory's update. computed per token as `sigmoid(beta_proj(x_t))`. range (0, 1).
- **erasure**: the optional subtraction term `a * b * k_t * (k_t^T * S_{t-1})` in the matrix memory's update. when present, every write subtracts the projection of the existing state onto the incoming key direction before adding the new key-value outer product, so writes targeting the same direction overwrite cleanly instead of stacking. when absent, the matrix memory is a pure accumulator: every write adds, nothing subtracts, and writes targeting the same direction stack indefinitely until decay removes them. the subtraction is applied in both execution paths when the toggle is true; which path runs depends on backend availability and sequence shape.
- **input encoding**: the function applied to keys, values, and queries before they enter the matrix memory's update. it remains a tunable part of the mechanism, but the current paid-run branch is not an encoding-only sweep because the earlier simulation work already closed that branch negative at the old operating point. the candidate space currently includes: identity (no transformation), three-level magnitude quantization with one global threshold per layer, three-level magnitude quantization with one threshold per dimension, fixed-fraction absolute-magnitude selection (keep the largest k of d entries, zero the rest), sign-only encoding (map to plus-or-minus one based on sign of the input, zero only when input is exactly zero), continuous bounded encoding (saturating nonlinearity with no zeros), and any new encoding that simulation work justifies adding.
- **long-distance recall task**: the evaluation in which a five-byte sequence is hidden inside a longer noise sequence and the architecture is asked to reproduce it after the noise. the architecture passes if it produces the correct sequence. tested at distances 256, 1024, 4096 tokens. for the current run-1 branch, passkey at 256 tokens is the hard gate; the longer distances are secondary diagnostics until the baseline clears that first gate.
- **marker-gated copy task**: the evaluation in which the architecture is asked to copy a marker-prefixed substring from earlier in the input. tested at lengths 256, 512, 1024, 2048 tokens. used as a secondary retrieval diagnostic for the current run-1 branch.
- **structure ratio**: the metric `mean ||probe_output|| / mean ||random_state_output||` measured over 32 random query vectors against the matrix memory's accumulated state after a fixed warmup. structure ratio near 1.0 means the matrix memory's state is statistically indistinguishable from random noise of the same Frobenius norm; structure ratio significantly below 1.0 (or above 1.0 with high pairwise cosine) means the state has stored content. used to diagnose whether the matrix memory has accumulated meaningful structure during training.
- **firing rate**: the fraction of dimensions in an encoded vector that are nonzero after the encoding is applied. depends on the chosen encoding and on the input distribution. measured per layer per step.
- **correction field**: the matrix memory operating mode in which values are projected from prediction residuals instead of raw hidden states. the state S becomes a map from query directions to prediction corrections: for any query q, `q^T @ S` returns the expected correction to the model's own prediction of the queried content. retrieval reconstructs as `prediction + correction`. described in `wiki/synthesis/correction_field_memory.md`.
- **prediction residual**: the vector `r_t = h_t - f(h_{t-1})` where h_t is the actual hidden state at timestep t and f is a lightweight learned prediction function. the residual is what the prediction function failed to anticipate. in the correction-field design, the value projection W_v operates on r_t instead of h_t.
- **surprise ratio**: the scalar `s_t = ||r_t||^2 / (||h_t||^2 + eps)` measuring how much of the hidden state's energy was unpredicted. range [0, 1]. s_t near 0 means the token was fully predictable; s_t near 1 means it was fully novel. used to modulate the write gate in the correction-field design.
- **self-compression**: the property of the correction-field design whereby the state's information content decreases as the model's prediction function improves during training. the compression ratio is not a hyperparameter but an emergent consequence of learning quality. early in training (bad prediction), the state stores approximately raw values; late in training (good prediction), the state stores only small corrections.
- **depth gradient**: the predicted phenomenon in the correction-field design where deeper layers' states carry less total information than shallower layers' states at convergence, because deeper layers' prediction heads explain more of the hidden-state variance. if confirmed, this creates a naturally emergent compression hierarchy within the existing 24-layer stack.
- **slot memory**: the proposed replacement substrate for the matrix memory. a per-head buffer of shape `(B, H, N, D)` where N is the slot count (default 64) and D is the per-slot content dimension (default 64). unlike the matrix memory, slots do not decay — they overwrite. reads use softmax addressing: `o = softmax(q @ slot_keys.transpose(-2,-1)) @ slot_values`. softmax restores exponential capacity per ramsauer et al. 2021, escaping the matrix memory's `O(d)` linear ceiling.
- **softmax addressing**: the read mechanism of the slot memory. given a query q and slot keys K of shape (N, D), the attention weights `softmax(q @ K^T / sqrt(D))` are used to produce a weighted sum of slot values. the softmax's exponential suppression isolates a single matching slot even under non-orthogonal keys, unlike linear `q^T S` readout which averages across all stored patterns with interference.
- **slot write policy**: the rule that decides which slot receives each new write. three options: (A) surprise-gated with least-recently-used allocation (uses the correction-field mechanism's prediction-residual surprise signal), (B) learned allocation scoring (a softmax over learned slot-key embeddings), (C) fixed hash allocation (baseline). option A is the default starting point.
- **output gate**: a sigmoid-bounded scalar per head that multiplies the slot memory's output before it is added to the residual stream. initialized near zero (via `gate_init = -4.0` so `sigmoid(-4) ≈ 0.018`). during training, the model learns to open the gate only for layers/heads where the slot memory's output actually helps. prevents noise injection from under-trained heads.
- **slot age / recency counter**: a per-slot counter that tracks how long since each slot was last read or written. used by the least-recently-used write policy to select which slot to overwrite when a new write arrives.
- **prosecutor pass**: a code review carried out by the `feature-dev:code-reviewer` agent (the non-claude-suffix variant). the review identifies findings ranked by priority. the rule is that every finding must be either fixed or explicitly waived in writing by the user before the related code is allowed to run on paid compute or to be published as a wiki article.

## the rule

`neuroloc/wiki/PROJECT_PLAN.md` is the canonical project state. it MUST be updated whenever any of the following happens:

- a training run starts
- a training run completes (with a one-paragraph entry in the prior runs index pointing to its run card)
- a hypothesis is added or falsified
- the experimental method changes
- a new project term enters the glossary
- a prior research question is answered (move it from open research questions to a paragraph somewhere relevant, with the answer)

failing to update this file is the same class of error as failing to run the prosecutor: both are blockers for paid compute launches. an outdated `PROJECT_PLAN.md` is a bug to fix, not noise to ignore.

the update history at the bottom of this file is append-only. every change to this file gets a new entry with date, author, and a one-line description of what changed and why.
